简介

SPI全称为：Service Provider Interface，是JDK内置的一种服务发现机制。

它是Java提供的一套用来被第三方实现或者扩展的API，可以用来启用框架扩展和替换组件，主要是被框架的开发人员使用，比如java.sql.Driver接口。

本质上来说，它也是典型基于面向接口编程思想设计的，这种设计思想想来大家应该也不陌生，在初学Java编程老师应该多次强调过。

SPITest.java

配置

在 resource 目录下，新建 META-INF/services 目录，再新建一个名为 com.Playwi0.spi.SPITest 的文件（文件名是你接口名称加上在当前项目的包名），内容写上该接口的所有扩展类。

调用

上面全部配置好后，接下来就是要去调用前面所提供的的服务。

ServiceLoader

Java 在 util 包里面提供了一个类 ServiceLoader 来实现调用。

调用其静态方法 load，传入接口类，会自动解析并返回配置中所有实现该接口的扩展类；最后遍历执行方法就可以了（ServiceLoader 实现 Iterable 接口）。

总结

Java SPI 确实具有非常强的扩展性，调用服务模块和服务模块之间进行了解耦，在实际业务中可以根据需要随意添加扩展类，但还是存在以下几处不足：

  • 不能按需加载，ServiceLoader 默认加载了所有扩展类。这里面可能大量是当前不需要的。
  • 即使加载所有扩展类都是我们需要的，获取的某个扩展类时比较麻烦，需要遍历。
  • ServiceLoader 类实例在多线程下是不安全的。

Dubbo SPI

虽然 Java SPI 具有很强的扩展性，但却有几处硬伤，不够灵活。针对这几个缺点，Dubbo SPI 在此基础上进行了增强，不仅增强其扩展性，同时还让其变得更加灵活。

原理

知道了怎么使用，来看看 Dubbo 是怎么实现的。

getExtensionLoader

对合法的可扩展接口，初始化一个 ExtensionLoader 放到缓存中

getExtension

从缓存取出指定扩展实例对象，没有就新建一个。不过扩展实例对象并不是直接放到缓存集合中，而是把它包装成一个 Holder 对象

getOrCreateHolder

createExtension

缓存没有就反射实例化指定扩展类，并进行依赖注入 injectExtension，最后执行生命周期方法 nitExtension。这里有 wrapperClasses，留着后面讲。

injectExtension

Dubbo 只支持 set 注入，利用反射来完成。

总结

前面分析了 Dubbo SPI 是如何实例化扩展类，并完成相应的依赖注入。这几处逻辑也并没有特别复杂的点。但这不是 Dubbo SPI 核心点。
